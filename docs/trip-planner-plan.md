# Планировщик поездок — целевая концепция и план развития

**Версия документа:** 2.0  
**Дата:** 2026-07-08  
**Статус:** целевая концепция (ревизия по ревью v1.0)

---

## 1. Позиционирование

### 1.1 Слоган (маркетинг, одна фраза)

> **Планируй мотопоездку дома, редактируй в пути, делись с друзьями.**

### 1.2 Определение продукта (внутреннее)

Планировщик — **план поездки с возможностью редактирования**, а не одноразовый маршрут навигатора. Три равнозначных сценария:

1. **Дома** — собрать многодневный план на большом экране, сохранить, перенести на телефон.
2. **В дороге** — применить сегодняшний день, править на остановке (ночёвка, объезд, заправка).
3. **Сообщество** — поделиться планом с другом или опубликовать шаблон (VK, Telegram, MAX, файл).

Источник истины — **JSON плана в приложении**. VK и мессенджеры — каналы распространения, не база данных.

*Связь с кодовой базой:* текущий MVP — «фаза 6» в `MASTERPLAN.md` (~45%: модель, storage, share, OSRM по сегменту).

## 2. MVP — явное определение

**Бета готова**, когда пользователь без помощи разработчика может:

| # | Критерий |
|---|----------|
| M1 | Создать или импортировать план на ПК |
| M2 | Выбрать мотоцикл из справочника (или один профиль по умолчанию) |
| M3 | Увидеть плановые заправки по расходу (если профиль задан) |
| M4 | Перенести план на телефон (файл **или** короткая облачная ссылка **или** `trip_pack` для компактного плана) |
| M5 | Нажать **«▶ Сегодня»** → OSRM → HUD с контекстом дня |
| M6 | Всё критичное работает **офлайн** после импорта (открытие плана, сегмент, локальный расчёт топлива) |

**Не входит в MVP:** редактор waypoints на карте, автослияние конфликтов, соцсеть (отзывы/рейтинги), POI-коридор, Markdown с погодой.

**Состав MVP в коде:** итерация **A-core** = A1 + A2 + A3 + A4 (см. §12).

## 3. Продуктовые гипотезы по итерациям

| Итерация | Гипотеза | Если не подтверждается |
|----------|----------|------------------------|
| **A-core** | Мотоциклистам нужен **план поездки**, а не только маршрутизатор | Остановить расширение планировщика; усилить HUD/навигацию |
| **A2** | Правки текста/ночёвки и конфликты версий нужны при двух устройствах | Отложить B; упростить до «одно устройство» |
| **B** | Пользователи **делятся** планами (≥15% планов с экспортом) | Не строить UGC-платформу; оставить файл + Web Share |
| **C** | Планы **редактируются в дороге** (≥2 ревизии у активных поездок) | Упростить до «только смотреть план» |
| **D** | Автоподбор POI/дней окупает сложность | Оставить ручное планирование + импорт GPX |

Метрики для проверки — §14. Решение о следующей итерации — по данным, не по чек-листу.

## 4. Сценарии

### 4.1 Дома (ПК, планшет)

| Шаг | Действие |
|-----|----------|
| 1 | Открыть moto-hud (браузер / PWA) |
| 2 | **Импортировать основу:** GPX, JSON плана, ссылку на готовый маршрут (Яндекс / Google / RideWithGPS — через GPX или точки), шаблон из сообщества |
| 3 | Создать или доработать план: старт, финиш, даты, ночёвки, лимит км/день |
| 4 | Выбрать **мотоцикл** из справочника профилей (KTM 1290, эндуро, турер…) — бак, расход, резерв, тип топлива |
| 5 | Система строит дни, **дистанции и плановые заправки** (если профиль задан) |
| 6 | Правка точек (в MVP — текстовые поля; позже — карта) |
| 7 | **Сохранить** локально + экспорт |
| 8 | **На телефон:** файл / облачная короткая ссылка / `trip_pack` (только компактные планы) / QR |

### 4.2 В дороге (телефон)

| Шаг | Действие |
|-----|----------|
| 1 | Импорт (ссылка, файл, «Избранное» VK/Telegram) |
| 2 | **«▶ Сегодня»** → OSRM → HUD |
| 3 | На остановке: сдвинуть ночёвку, добавить точку, **«Заправился»** (литры, одометр) |
| 4 | **Пересобрать оставшиеся дни** при укорочении сегодня (дождь, усталость, «остался в красивом месте») — предложить новые дистанции/ночёвки, не автоматом в MVP |
| 5 | Отправить обновление: JSON / короткая ссылка / share |
| 6 | HUD: день N, до ночёвки, до заправки; при расхождении с ГдеБЕНЗ — **факт важнее плана** (см. §6.5) |

**Офлайн:** правки пишутся локально; при появлении сети — экспорт или правило «последняя ревизия по `updatedAt` побеждает» (MVP).

### 4.3 Друзья и сообщество

**MVP (итерация B):** нативный share URL/файла в VK, Telegram, MAX; форк с указанием автора; каталог 10–20 шаблонов на GitHub Pages.

**Не в MVP (итерации A–C):**

- Отзывы, рейтинги, «проехал в июле 2026»
- Краудсорс ночёвок/АЗС в комментариях VK
- Поиск и модерация 100+ UGC-планов

Это **отдельная фаза «соцсеть»** — явно вне скоупа до подтверждения гипотезы B.

## 5. Принципы проектирования

1. **Offline-first в поездке** — открытие плана, сегмент, локальный расчёт топлива, пересборка дней по локальным данным; сеть только для шаринга и загрузки новых шаблонов.
2. **JSON — канон**; Markdown — для людей; `trip_pack` — вспомогательный канал для **компактных** планов.
3. **Конфликты версий — только вручную в MVP** (взять телефон / ПК / копия). Автослияние по дням **не делаем** до отдельного проектирования (конфликты внутри дня часты).
4. **Сообщество с первого дня в модели** — `author`, `sharePolicy`, `forkOf`; UI «Поделиться» рядом с «Сохранить».
5. **Топливо — плановый слой**; **ГдеБЕНЗ/краудсорс — факт в HUD**. При расхождении: уведомление *«плановая АЗС на 320 км недоступна, ближайшая с топливом на 340 км»*.
6. **В движении — минимум ввода**; карта и сложные правки — на остановке.
7. **Приватность по умолчанию** — план локально; публикация только явным действием; в share уходит весь маршрут (старт = часто дом).
8. **Устойчивость к потере связи** — неделя в горах без интернета не должна ломать сценарий «▶ Сегодня».

## 6. Модель данных

Цепочка: `Trip → Day → Variant → Segment` (`rtext` = waypoints).

### 6.1 Справочник мотоциклов (новый модуль `bike-profile.js`)

Хранится в настройках пользователя (localStorage / IDB), не внутри одного плана:

```json
{
  "id": "bike_ktm_1290",
  "name": "KTM 1290 Super Adventure",
  "tankLiters": 22,
  "reserveKm": 40,
  "fuelType": "95",
  "consumptionL100": {
    "default": 5.2,
    "highway": 5.0,
    "city": 6.5,
    "mountain": 6.5,
    "offroad": 8.0
  }
}
```

В плане — **ссылка + снимок** (чтобы старые планы не ломались при удалении профиля):

```json
{
  "meta": {
    "bikeProfileId": "bike_ktm_1290",
    "bikeProfileSnapshot": { "tankLiters": 22, "consumptionL100": { "default": 5.2, ... }, "fuelType": "95", "reserveKm": 40 },
    "author": { "name": "Илья", "vkUrl": null },
    "sharePolicy": "private",
    "forkOf": null,
    "revision": 3,
    "updatedAt": "2026-07-08T10:00:00Z",
    "deviceLastEdit": "phone",
    "tags": ["кавказ", "14дней"],
    "regionHint": "central_ru | north | caucasus | siberia"
  }
}
```

**Поведение UI без профиля:** блок «⛽ Топливо» **скрыт**, не показывать «0 заправок» / «неизвестно».

### 6.2 День и сегмент (расширения)

| Поле | Назначение |
|------|------------|
| `day.date` | Календарная дата |
| `variant.night` | Ночёвка (текст + lat/lon) |
| `variant.weatherNote` | Заметка / ссылка на прогноз (экспорт в MD) |
| `variant.alt` | Запасной вариант дня (трасса в объезд при дожде) |
| `segment.plannedKm` | OSRM или аудит |
| `segment.roadMix` | Доли highway/city/mountain/offroad — для расхода |
| `segment.fuelPlan` | Плановые заправки (§7) |
| `segment.status` | `planned` \| `active` \| `done` |
| `poi[]` | POI дня (фаза D) |

### 6.3 Версионирование

- `TRIP_MODEL_REV = 2` при `bikeProfileSnapshot` и `revision`.
- Миграция v1→v2: без профиля — топливный UI скрыт.

## 7. Топливное планирование

### 7.1 Запас хода

```
rangeKm = (tankLiters / consumptionEffective) * 100 - reserveKm
```

`consumptionEffective` — взвешенный по `segment.roadMix` или `default`, если OSRM-классификации ещё нет.

### 7.2 Литры на сегмент

```
fuelLiters = Σ (km_i * consumptionL100[roadType_i] / 100)
```

Один средний расход даёт погрешность до 20–30% на смешанных маршрутах — в документации и UI указывать «оценка».

### 7.3 Алгоритм плановых заправок (жадный по АЗС)

**Не** «70–80% запаса хода» фиксированно — зависит от плотности АЗС.

1. Загрузить АЗС в коридоре маршрута (`ensureFuelStations` / OSM).
2. Отфильтровать по `fuelType` (OSM: `fuel:octane_95=yes` и т.п.).
3. Пройти маршрут слева направо: на каждом шаге ехать до **последней АЗС**, до которой хватает `(запас − резерв)`; при необходимости заправиться **раньше** в разреженных регионах (Карелия, Алтай, Дальний Восток — `regionHint` снижает порог риска).
4. Если между двумя обязательными точками нет подходящих АЗС — warning в `fuelPlan.warnings`.

**Развитие (не MVP):** совмещение заправки с привалом (АЗС + `amenity=cafe`/туалет).

### 7.4 «Заправился» и калибровка расхода

Поля события: `liters`, `odometerKm` (опционально). По разнице литры/км — предложение обновить `consumptionL100.default` в профиле мото.

### 7.5 Приоритет слоёв в HUD

| Приоритет | Источник |
|-----------|----------|
| 1 | Краудсорс / ГдеБЕНЗ (факт) |
| 2 | Плановая АЗС из `fuelPlan` |
| 3 | Ближайшая OSM без статуса |

Сообщение пользователю при смене приоритета — обязательно.

### 7.6 Подшаги разработки A2 (топливо)

1. Справочник мото (один профиль в MVP-бете → список профилей).
2. `rangeKm` без stops.
3. Stops: жадный алгоритм + `ensureFuelStations`.
4. UI-блок в карточке дня.

Оценка: **1–1,5 недели** при вечерней разработке на подшаги 1–4.

## 8. Синхронизация ПК ↔ телефон

### 8.1 Каналы (реалистично)

| Канал | Покрытие планов | Примечание |
|-------|-----------------|------------|
| **JSON-файл** | 100% | Основной для полных планов |
| **Облако: короткая ссылка** | 100% | `POST` JSON → `motohud.link/t/abcd` (Cloudflare Worker, 1–2 дня backend) |
| **`trip_pack` в URL** | **~30–40%** | Только базовые планы без POI и без тяжёлых снапшотов; лимит ~6–7 KB gzip |
| **QR** | Как `trip_pack` | QR с 20+ KB ненадёжен; для полных планов — QR на **короткую ссылку** |
| **VK / Telegram / MAX** | 100% | Файл или URL с OG-страницей |

**OG-страница** (`motohud.link/trip/abcd`): title, дни, км, превью карты — иначе VK Share без превью. Это тот же Worker, что и короткая ссылка.

### 8.2 Конфликты (MVP)

1. Сравнить `revision` / `updatedAt`.
2. Показать diff (дни, ночёвки, waypoints).
3. Выбор: **телефон** | **ПК** | **сохранить как копию**.
4. **Без автослияния** в MVP.

**Офлайн-очередь (MVP):** правки на телефоне без сети → при импорте более новой копии — правило **last-write-wins по `updatedAt`** с предупреждением.

### 8.3 User-outcome итерации A-core

> *Пользователь создаёт план дома, переносит на телефон, нажимает «▶ Сегодня» и едет с контекстом дня.*

### 9.2 VK: фазы и риски

| Фаза | Содержание | Сроки (реалистично) |
|------|------------|---------------------|
| B1 | Share URL + файл, каталог шаблонов | 2–3 нед. (side-project ×2) |

### 9.3 Сообщество: масштаб

- **MVP:** категория только **«маршруты»**; curated JSON на GitHub Pages (10–20 планов).
- **>50 планов:** нужен процесс модерации и приёмки — пересмотр архитектуры.
- **Правила сообщества:** запрет публикации чужих адресов без согласия, жалобы, снятие поста (§17).

### 9.4 User-outcome итерации B

> *Пользователь публикует план в VK/Telegram; друг сохраняет к себе с указанием автора.*

## 14. Метрики

### 14.1 Операционные (MVP)

| Метрика | Цель |
|---------|------|
| ПК → телефон, первый успех | ≤ **5 мин** (потом улучшать до 3) |
| Сдвинуть ночёвку | ≤ 5 тапов |
| «Заправился» (литры) | ≤ 4 тапа |
| Объезд с картой | ≤ 10 тапов (отдельная метрика) |
| Дни с km > range и ≥1 заправкой в плане | 100% (если профиль задан) |
| Потеря плана при переносе | 0 при JSON/облаке |

### 14.2 Продуктовые

| Метрика | Зачем |
|---------|-------|
| **Retention 7d** | Вернулся к плану через неделю |
| **Доля планов с ≥1 днём `done`** | План реально применяют, не только создают |
| **Среднее число `revision`** | «Живой» документ (цель >2 у активных поездок) |
| **Доля планов с экспортом/share** | Виральность (гипотеза B) |

### 14.3 Аналитика (opt-in)

Минимальный локальный/анонимный счётчик (без GA по умолчанию): применений «▶ Сегодня», экспортов, сдвигов ночёвки. Dashboard для разработчика — агрегаты раз в релиз.

## 15. Полевые тесты

Перед B и после A-core:

- **3–5 мотоциклистов** вне команды разработки.
- Типы: одиночка, пара, группа; выходные / 3–7 дней / дальняк.
- Регионы: центр РФ, север (редкие АЗС), горы.
- Анкета + короткое интервью после поездки.
- Повторять **раз в квартал**.

| `trip_pack` слишком мал для большинства планов | Облако `motohud.link` как основной канал; trip_pack — «лёгкие планы» |
| Топливный расчёт ±30% → негатив | UI: «оценка»; калибровка по «Заправился»; жадный алгоритм по АЗС |
| Пользователи не шарят | Не строить соцсеть; file-first |
| Конфликты версий путают | Только ручное разрешение + копия; LWW с предупреждением |
| OSM/АЗС пустые в регионе | Warnings + ручная точка заправки |
| Side-project затягивается | Жёсткий A-core без A5–A7 до полевого теста |

## 17. Сообщество: правила и модерация

- Публикация — согласие на передачу маршрута получателям.
- Не публиковать точные координаты частных домов без согласия владельца.
- Жалоба → снятие из curated-каталога (ручная модерация на этапе <50 планов).
- Платная реклама гостиниц/АЗС в планах — вне скоупа MVP.

## 19. Региональная специфика

Поле `regionHint` в плане и профиле пользователя влияет на:

- пороги риска в топливном алгоритме (плотность АЗС);
- дефолтные лимиты км/день в мастере;
- подсказки в онбординге (OSM в Карелии vs М4).

Полная локализация (en) — фаза 7 `MASTERPLAN`.

## 21. Вне скоупа (явно)

- Обязательный аккаунт / автосинхронизация без действия пользователя.
- **Соцсеть внутри приложения:** отзывы, рейтинги, лента, комментарии к планам (до отдельного решения после B).
- Автослияние конфликтов версий (до v2 sync).
- ИИ-маршрут по описанию («хочу серпантины 5 дней»).
- Парсинг Яндекс.Карт / пробки в офлайн.
- Фото/видео-отчёты в плане.
- Интеграции: страховки, отели, шиномонтажи (комиссия).
- Границы / визы.
- VK как единственная БД.
- Premium/монетизация планировщика — до подтверждения спроса.

## 22. Следующий шаг (A-core)

Порядок разработки:

1. **`bike-profile.js`** — один профиль по умолчанию → список.
2. **`trip-fuel.js`** — только `rangeKm` + warnings без stops.
3. **Stops** — жадный алгоритм + `ensureFuelStations` + фильтр `fuelType`.
4. **UI** — блок «⛽» в карточке дня (скрыт без профиля).
5. **«▶ Сегодня»** + `tripContext`.
6. **«На телефон»** — JSON + Worker короткой ссылки (параллельно можно file-only, Worker — следом).
7. **Полевой тест** на 1 короткой поездке до расширения A2.
