# Мото-ИЛС (moto-hud) — мастер-план развития

> Живой документ. Обновляется по мере развития проекта.
> Последняя ревизия: 2026-07-05 (доп.).

---

## 1. Видение продукта

**Мото-ИЛС** — навигатор-«стеклянная кабина» для мотоцикла: крупный, читаемый через визор HUD
(манёвр, дистанция, скорость, прогноз-дорожка, курс), голосовые подсказки в гарнитуру шлема,
надёжная работа **без сети** и в фоне.

Ключевые отличия от обычного навигатора:
- считывается за долю секунды на скорости и через тонированный визор;
- голос манёвров приходит в гарнитуру шлема без «танцев с бубном»;
- работает офлайн (карты + маршрут), потому что связь в горах и на трассе часто отсутствует;
- учитывает специфику мото: извилистость дороги, камеры «в спину», крупная моторика в перчатках;
- уживается с другими приложениями (Яндекс Навигатор, антирадар, музыка), а не воюет с ними.

**Целевая платформа №1 — Android** (аудитория мото + раггед-телефоны). iOS — позже.

## 2. Текущее состояние (baseline, честно)

**Есть:**
- Единое веб-приложение `index.html` + эмулятор `sim.html`.
- HUD на CSS Grid: статус-строка, крупная параметрическая стрелка манёвра, улица, дистанция,
  крупная скорость, прогноз-дорожка (3D-проекция), опциональный компас N/E/S/W.
- Адаптивная раскладка (портрет / 3 колонки в альбоме), полноэкранный режим.
- Плавная отрисовка: интерполяция позиции между GPS-фиксами (dead reckoning) + `requestAnimationFrame`,
  батч-сборка SVG, snap по длине дуги (`RouteGeometry`).
- **Прогноз-дорожка v2:** единая `RouteGeometry` (Float64Array dense, snap по `s`, Frenet, triangle strip),
  профиль высот, числовые метки уклона — см. [`docs/ROADPATH-PLAN.md`](docs/ROADPATH-PLAN.md),
  реализация: [`docs/route-geometry.md`](docs/route-geometry.md).
- Веб-API: `geolocation.watchPosition`, `speechSynthesis`, `wakeLock`, Fullscreen.
- Эмулятор движения с маршрутом из поворотов разной крутизны и разворотом.
- Деплой на GitHub Pages через GitHub Actions.

**Нет (разрывы до «настоящего мотонавигатора»):**
- ❌ Офлайн: **всё онлайн** — маршрут (OSRM), камеры/POI (Overpass), поиск (Nominatim).
- ✔ Устанавливаемость: `manifest.json` + service worker (Фаза 1 закрыта).
- ❌ Фоновый GPS: в браузере умирает через 30–60 с после сворачивания.
- ❌ Гарантированный голос в гарнитуру: нет audio focus / управления BT-профилем.
- ❌ Стабильный курс на малой скорости: heading только из GPS, без сшивки с сенсорами.

## 3. Стратегические принципы

1. **Два независимых трека.** Развиваем параллельно:
   - **(A) Платформа/оболочка** — как приложение доставляется и получает доступ к железу.
   - **(B) Карты + роутинг** — что делает его навигатором (и офлайн).
   Для мотонавигатора (B) важнее (A) и почти от него не зависит.
2. **Ничего не выбрасываем зря.** Порядок: PWA → Capacitor (та же веб-сборка) → точечный натив.
   Переписывание на Flutter/натив — только если продукт «взлетит».
3. **Соло-темп.** Каждая фаза даёт работающий результат и не блокирует следующую.
4. **Безопасность важнее фич.** Читаемость, голос, отсутствие отвлекающих действий — приоритет №1.
   В движении — только крупная моторика и короткие голосовые ответы («да/нет»); сложный ввод — на остановке.
5. **Тестируемость.** Любая фича проверяется в `sim.html` до выезда. **Основные фикстуры — реальные
   GPX-треки**, проигрываемые в симуляторе (регресс для dead reckoning, логики упреждения подсказок,
   стабилизации курса). Синтетический маршрут сохраняем, но как **дополнение** к реальным записям.

## 4. Решения по стеку (с обоснованием)

| Слой | Решение | Почему |
|------|---------|--------|
| Оболочка | **Capacitor** (после PWA-обвязки) | 90% текущего кода сохраняется; графика лёгкая (SVG+rAF ~0.47 мс/кадр) — минус WebView не важен; публикация в стор; нативные плагины точечно |
| UI/рендер | Остаётся **HTML + SVG + rAF** | Уже плавно и дёшево; выгода Flutter CustomPainter для нас почти нулевая |
| Карты (офлайн) | **MapLibre GL Native** + тайлы **PMTiles** | Свободный, векторные тайлы, биндинги под все оболочки, обёртки под Capacitor |
| Роутинг (офлайн) | **Valhalla / GraphHopper / BRouter** (спайк-выбор, Фаза 4) | Свободные, встраиваемые; мото-профили с извилистостью, surface/smoothness, round-trip; финальный выбор — по спайку |
| Голос | `speechSynthesis` сейчас → **нативный TTS + кооперативный audio focus** в Capacitor | Гарантия доставки подсказки в гарнитуру, приглушение музыки/звонка, без наложения на чужой голос |
| Гарнитура | Нативный BT-плагин (HFP/A2DP, audio focus) | В вебе управление профилем невозможно — это мото-критично |
| Натив Kotlin | Отложен, точечно | Только под фон/сенсоры/Android Auto, когда появится потребность продукта |

### Фаза 1 — PWA-обвязка ✅ (сделано)
**Цель:** устанавливается с иконкой, интерфейс работает без сети.
- [x] `manifest.json`: имя, иконка (SVG `any` + `maskable`), `display: fullscreen`, ориентация, тема.
- [x] Service worker (`sw.js`): precache оболочки, версионируемый кеш, cache-first/stale-while-revalidate.
- [x] Мета-теги PWA, регистрация SW (пропуск в режиме эмулятора), offline-fallback интерфейса.
- [x] Кеш последней поездки (маршрут + камеры) — MVP на **localStorage**, переигрывание при офлайне
  к тому же финишу. **Миграция на IndexedDB — в Фазе 4** (больший объём, структурированное хранение).
**Готово, когда:** приложение ставится на домашний экран Android, открывается офлайн (интерфейс),
последний маршрут доступен без сети. ✔ Проверено локально: SW `activated`, кеш оболочки наполнен,
оболочка отдаётся из кеша офлайн.
**Заметки:** icon в SVG (без бинарников); для «идеальных» иконок стора позже добавить PNG 192/512.
iOS-ограничения PWA — на этой фазе не целимся в iOS.

**Цель:** APK/AAB из текущего кода + надёжный фоновый GPS.
- [x] **Модуляризация `index.html`** — разбить монолит на модули (рендер, GPS, маршрут, голос, UI).
  **Первая задача, до упаковки в Capacitor** — иначе дальнейшая поддержка и нативные вставки усложняются.
- [x] Инициализация Capacitor, сборка Android-проекта из веб-бандла (`npm run build` → `www/`, `npx cap sync`).
- [x] Плагин фоновой геолокации (`@capacitor-community/background-geolocation` при активной навигации).
- [x] Foreground-service с уведомлением «навигация активна», **`foregroundServiceType="location"`** (через плагин).
- [x] **Декларация типа foreground-сервиса для Android 14+** (`FOREGROUND_SERVICE_LOCATION`, `POST_NOTIFICATIONS` в манифесте).
- [x] **Экран/wake-lock нативно** — `@capacitor-community/keep-awake` + Screen Wake Lock API (PWA); `js/wake-lock.js`.
- [x] Разрешения: точная геолокация на переднем плане; онбординг «отключите оптимизацию батареи / автозапуск» (подсказка в «Как пользоваться» на Android).
- [ ] **OEM-матрица тестов** (≥ 3 вендоров — см. [`docs/oem-gps-matrix.md`](docs/oem-gps-matrix.md)): проверка выживания фонового сервиса.
- [ ] Внутренний тестовый трек Play Console (closed testing) — чек-лист [`docs/play-console-closed-testing.md`](docs/play-console-closed-testing.md).

на устройствах ≥ 3 вендоров.
**Риск:** выживание фонового сервиса на прошивках вендоров (корень — OEM, а не сам плагин) —
прототипировать в первую очередь, до остальных фич.

### Фаза 3 — Голос в гарнитуру + стабильный курс (критерии готовности, без жёстких сроков)
**Цель:** подсказки гарантированно звучат в шлем и не наслаиваются на чужой голос; курс не «плавает».
- [x] **Нативный TTS (MVP)** — `@capacitor-community/text-to-speech` на Android; очередь фраз, антиспам (`js/voice.js`).
- [ ] **Кооперативный audio focus** (см. разд. 6): ducking музыки на время фразы,
  вклинивание при звонке, **очередь при занятом фокусе** (говорит Стрелка/Яндекс — откладываем).
- [x] **Офлайн-TTS (MVP):** `js/tts-health.js` — проверка ru-RU / офлайн-голосов, баннер в setup,
  `TextToSpeech.openInstall()` на Android. Полная проверка в режиме «в самолёте» — на устройстве.
- [ ] Проверка A2DP/HFP-маршрутизации на реальной гарнитуре шлема.
- [x] **Курс: DeviceOrientation + GPS (MVP)** — `js/heading.js`, сшивка на малой скорости; полный **ROTATION_VECTOR** нативно — далее.
- [x] **UX калибровки** магнитометра (жест «восьмёрка») + автоматический **fallback на GPS-курс**
  при подозрении на помехи. **MVP:** кнопка в опциях, детектор расхождения GPS/сенсор, предупреждение в HUD.
- [x] **Упреждение подсказок по времени, а не по метрам:** момент озвучки = f(скорость) — за N секунд
  хода до манёвра (на трассе предупреждать сильно раньше, чем в городе).
- [ ] Приоритезация и антиспам подсказок (не повторять, не наслаивать).
**Готово, когда:** голос манёвра слышен в гарнитуре поверх музыки и не накладывается на сторонний
навигатор; курс стабилен на стоянке и в пробке; при магнитных помехах заметен fallback на GPS-курс.
**Риск:** **магнитные помехи от рамы/электрики мотоцикла** искажают магнитометр → курс «уплывает».
Мера: сшивка с GPS-курсом, автоматический fallback и UX калибровки.

### Фаза 4 — Офлайн-карты и офлайн-роутинг (⭐ ключевая ценность; сроки — после спайка)
**Цель:** навигатор работает без интернета.
- [ ] **Спайк: сравнение офлайн-роутеров** — **Valhalla vs GraphHopper vs BRouter**. BRouter лёгкий,
  давно проверен на Android, профили настраиваемы. Критерии выбора: размер графа, скорость расчёта,
  качество мото-профиля (извилистость), **наличие высот из коробки**, **round-trip из коробки**,
  **гибкость кастомных профилей** (surface/smoothness, извилистость, scenic/viewpoint),
  простота встраивания в Capacitor (см. [`docs/router-spike.md`](docs/router-spike.md)).
  **CLI:** `npm run spike:routers` — OSRM vs BRouter на демо-маршруте.
- [ ] MapLibre GL с векторными тайлами (PMTiles); подложка под прогноз-дорожку/обзорную карту.
- [ ] **Пайплайн данных PMTiles** — генерация тайлов (**Planetiler**) из OSM, нарезка по регионам,
  хостинг, регламент обновления. **Отдельная инфраструктурная работа — учитывать в сроках отдельно
  от клиентской части.**
- [ ] Менеджер регионов: скачать/удалить область, показать размер и покрытие.
- [ ] Выбранный офлайн-роутер, граф региона, **мото-профиль** с **настраиваемыми весами** по тегам OSM
  **`surface` / `smoothness`**: штраф *или* бонус в зависимости от предпочтения покрытия (см. Фазу 5).
  По умолчанию для шоссейника — «не заведи на грейдер»; для эндuro/приключенческого — наоборот,
  «хочу грейдер, грязь». Жёсткого запрета типов дорог нет — только веса профиля.
- [ ] Абстракция маршрутизации: онлайн-OSRM ↔ офлайн-роутер за единым интерфейсом; единый переключатель
  профилей поездки («быстро» / «извилисто» / «красиво»).
- [ ] Пересчёт маршрута офлайн при сходе с трассы (цель ≤ 3 с).
- [ ] **Миграция кеша поездок на IndexedDB** (с localStorage): больший объём, структурированное
  хранение маршрутов и метаданных тайлов/регионов.
**Готово, когда:** маршрут строится и пересчитывается в авиарежиме на скачанном регионе.
**Заметки:** голосовой поиск адресов офлайн — сложно; допускаем онлайн-геокодер для поиска.
Офлайн-навигация нема без офлайн-голосов TTS (см. Фазу 3).
**Ограничение (честно):** **пробки учитываются только в онлайн-режиме планирования** (если доступен
внешний источник). В **офлайн-навигации пробок нет и не обещаем**: открытых real-time источников
по РФ нет, данные Яндекса лицензионно закрыты (разд. 6).

### Фаза 5 — Мото-функции (итеративно)
**Цель:** то, ради чего мотоциклист выберет именно это.
- [ ] **Камеры офлайн.** Источники: **OSM** (ODbL, обязательна атрибуция) и **официальные открытые
  данные** — карта **ФВФ ГИБДД**, **data.mos.ru** и региональные порталы открытых данных. Предзагрузка
  по региону (вместо Overpass в реальном времени). **Не используем** проприетарные базы (Стрелка и т.п.).
  **Мобильные засады не покрываем** — для них пользователь запускает антирадар параллельно (разд. 6).
  Собственный краудсорсинг камер — в бэклог.
- [ ] **Запись треков.** Append-only **SQLite / JSONL**, частота 1 Гц: координаты, скорость, курс,
  точность. Экспорт в **GPX 1.1** через **SAF** (Storage Access Framework). Упрощение
  **Douglas–Peucker** — **только для отображения**, не для хранения (храним сырьё). Приватность:
  **локально по умолчанию**, ничего не отправляется без явного действия пользователя.
- [ ] Статистика поездки: макс/сред. скорость, дистанция, время в движении, углы.
- [x] **Топливный ассистент (кнопка ⛽ на HUD).** Цикл на одну кнопку для решения на скорости
  120 км/ч: 1-й тап — ближайшая АЗС **по маршруту** (в коридоре впереди, без пересчёта);
  2-й тап — ближайшая АЗС **вообще** + перестроение маршрута к ней как к финишу; 3-й тап — отмена
  и возврат к исходному финишу. Крупная центральная панель + озвучка. Символы ⛽ и расстояние на
  прогноз-дорожке (второстепенно). Источник точек — **OSM Overpass `amenity=fuel`**; наличие топлива —
  **ГдеБЕНЗ** (см. ниже), цвет символа = статус (есть/очередь/мало/нет), иначе нейтральный.
- [ ] **POI по интересам (офлайн по скачанным регионам).** Категории:
  - **Заправки** (`amenity=fuel`) — частично реализовано (⛽ на HUD);
  - **Еда/кафе** (`amenity=restaurant`, `cafe`, `fast_food`, …);
  - **Кемпинги и ночёвки** (`tourism=camp_site`, `tourism=hotel|guest_house|motel`, …);
  - **Мотосервисы** (`shop=motorcycle`, `shop=motorcycle_repair`, …);
  - **Достопримечательности** с подкатегориями: виды/viewpoint, монастыри/храмы
    (`historic=monastery|church`, `amenity=place_of_worship`), музеи (`tourism=museum`),
    природные объекты (`natural=*`, `tourism=attraction`).
  **Фильтр:** показываются только категории из **профиля интересов** пользователя, не все разом.
- [ ] **Профили поездки** — один переключатель, три конфигурации весов роутера:
  - **«Быстро»** — минимум detour, приоритет скоростных/магистральных;
  - **«Извилисто»** — максимум серпантинов, штраф за прямолинейность;
  - **«Красиво»** — бонус за scenic/viewpoint, мягкий обход residential, баланс время/виды.
- [ ] **Предпочтение покрытия** (ортогонально профилю поездки, в настройках мото):
  - **«Асфальт»** — штраф за грейдер, грунт, `track`/`path` без твёрдого покрытия (шоссейник);
  - **«Смешанный»** — нейтрально, едем куда ведёт маршрут;
  - **«Бездорожье»** — бонус за грейдер, грунт, `surface=unpaved|gravel|ground`, `smoothness=bad|horrible`
    («хочу в грязь» — эндuro, приключенческий, кросс). Мотоциклы разные — профиль выбирает пользователь.
  Веса `surface`/`smoothness` в роутере **двунаправленные**, не односторонний запрет.
- [ ] **Голосовые команды POI (без LLM).** Фиксированный набор: «заправка», «кофе», «сервис» →
  ближайший POI выбранной категории **вдоль маршрута** (привязка к дуговой координате `s`
  из `RouteGeometry`) → озвучка + подтверждение «да/нет» → при «да» пересчёт маршрута.
  Работает офлайн по скачанным POI. В движении — только «да/нет» (принцип №4).
- [ ] **Круговой маршрут (round-trip):** «N часов / M км от текущей точки, поизвилистее» → петля
  через **штатную функцию роутера** (round-trip из коробки или эквивалент через waypoints).
- [ ] Опц.: температура/давление, угол наклона (если есть сенсоры), время до заката.
**Готово, когда:** каждая фича проверена в симуляторе и на выезде, не отвлекает от дороги.
**Заметка:** фичи Фазы 5+ проектировать с учётом деления free/premium (см. Фазу 8).

### Фаза 6 — Планировщик многодневных путешествий
**Цель:** спланировать поездку на несколько дней онлайн, проехать офлайн (регионы скачиваются заранее).
**Принцип:** планирование онлайн (геокодер, POI, высоты), поездка офлайн (карты + граф + POI в регионах).

#### Ввод
- [ ] Старт, финиш, даты поездки.
- [ ] Выбор **интересных категорий POI** (из профиля интересов, см. Фазу 5).
- [ ] Для каждой категории — **бюджет отклонения** (км и/или минуты: насколько готов уйти с маршрута
  ради точки этой категории).
- [ ] **Лимит км/день**, **профиль дорог** («быстро» / «извилисто» / «красиво») и **предпочтение покрытия**
  («Асфальт» / «Смешанный» / «Бездорожье», см. Фазу 5).

#### Выход
- [ ] Разбивка на **дни** с дневными перегонами.
- [ ] Предложение **точек ночёвки** в конце каждого дня.
- [ ] Список **POI по дням** с координатами.
- [ ] Сводная таблица «**день / маршрут / ночёвка**».

#### Модель данных: `Trip → Day → Segment → Waypoint/POI`
- [ ] **«База»** — ночёвка на несколько ночей в одной точке; дневные выезды из неё.
- [ ] **«Радиалка»** — дневной кольцевой выезд из базы с возвратом (частный случай round-trip,
  старт = финиш = база).
- [ ] **«Длинный перегон»** — сегмент с флагом «можно разделить на два дня».
- [ ] **Точки вне мото-достижимости** — пометка «не для мотоцикла»; в маршрут не включаются,
  показываются как справочные.

#### Алгоритм подбора POI
- [ ] Коридор вдоль **базового маршрута**; точка включается только если отклонение укладывается
  в **бюджет её категории**.
- [ ] При конфликте бюджетов — приоритизация по рейтингу/явному выбору пользователя.
- [ ] Зафиксировано в плане: задача — вариант **orienteering problem**; точного решения не требуется,
  достаточно **жадной эвристики** с **ручной докруткой** (drag точек, добавление/удаление).

#### Ручное редактирование (обязательно)
- [ ] Планировщик **предлагает**, пользователь **правит**: перенос POI между днями, смена ночёвок,
  фиксация «обязательных» точек.

#### Экспорт / импорт
- [ ] **GPX по дням** (отдельный файл или сегмент на день).
- [ ] Передача точек в **Яндекс Карты/Навигатор** интентом `yandexnavi://` (разд. 6).
- [ ] **Шаринг плана текстом** (день / маршрут / точки / ночёвка).

#### Итерации
- [ ] **MVP (первая итерация):** один базовый маршрут + **коридорный подбор POI** + разбивка по дням
  по лимиту км. Без баз/радиалок и без бюджетов по категориям.
- [ ] **Вторая итерация:** базы, радиалки, бюджеты отклонения по категориям, «длинный перегон»,
  точки вне мото-достижимости.

#### HUD-режим по плану
- [ ] При езде по плану: «**день N**, до ночёвки **X км**»; **следующий POI дня** — промежуточная цель
  (привязка к `s` на дневном сегменте).

**Готово, когда:** MVP-план из двух точек и лимита км/день собирается онлайн, скачивается в регион,
едется офлайн с отображением дня и ночёвки на HUD; пользователь может вручную поправить план до выезда.

## 6. Интеграция и сосуществование с другими приложениями

Мотоциклист редко пользуется одним приложением: параллельно могут работать Яндекс Навигатор/Карты,
антирадар (Стрелка и т.п.), музыка. Мото-ИЛС должен вести себя **кооперативно**, а не воевать за ресурсы.

- **Кооперативный audio focus.** Голосовую подсказку сопровождать запросом
  **`AUDIOFOCUS_GAIN_TRANSIENT_MAY_DUCK`** только на время фразы, затем сразу отпускать фокус.
  Если фокус занят (говорит другой навигатор/антирадар) — **откладывать** подсказку в очередь,
  **не наслаивать** поверх чужого голоса.
- **Режим «HUD-оверлей» (`SYSTEM_ALERT_WINDOW`).** Компактный HUD (скорость, курс, ближайший манёвр)
  поверх стороннего навигатора (напр. Яндекс) — как отдельная фича: маршрут ведёт Яндекс, а Мото-ИЛС
  даёт крупный, читаемый через визор слой. Требует разрешения «поверх других окон».
- **Режим «спидометр + камеры» без активного маршрута.** Без построенного маршрута приложение работает
  как крупный спидометр с предупреждением о камерах «в спину».
- **Открытие маршрута в Яндекс Навигаторе.** Кнопка запуска официального интента **`yandexnavi://`**
  (документированная схема) — легальный способ передать точку/маршрут во внешний навигатор.
  Используется также для **экспорта точек из планировщика путешествий** (Фаза 6).
- **Лицензионное ограничение (жёстко).** Данные Яндекса (карты, **пробки**, камеры) **извлекать/парсить
  нельзя** — запрещено лицензией. Yandex MapKit SDK допустим **только как альтернатива всего
  картографического стека** (вместо MapLibre+PMTiles), но **не как источник данных** поверх нашей карты.
  **Пробки в офлайн-навигации не обещаем** (см. Фазу 4).

## 9. Метрики успеха (измеримые)

- **Читаемость:** ключевая информация считывается ≤ 0.5 с (тест на выезде, целевой).
- **Батарея:** ≤ **10% в час** активной навигации на референс-устройстве.
- **Надёжность фона:** 2 ч поездки, **0 разрывов GPS длиннее 5 с**, на **3 устройствах разных вендоров**.
- **Голос:** подсказка озвучена **не позже чем за N секунд хода** до манёвра (время, не метры);
  100% манёвров без потери на музыке/чужом голосе.
- **Офлайн-пересчёт маршрута:** ≤ **3 с** на скачанном регионе.

## Разбор ГдеБЕНЗ (gdebenz.ru) — референс и источник данных о топливе

Проанализирован 2026-07-04. Краудсорсинговая карта наличия топлива на АЗС России.

- **Карта:** растровые тайлы **CARTO Voyager** (на данных OSM), самопроксируются
  через `cdn.gdebenz.ru/basetiles/rastertiles/voyager/{z}/{x}/{y}.png`. Полезно как **растровый
  fallback** к нашему вектору MapLibre+PMTiles (Фаза 4), если векторный стек будет тяжёл.
- **API наличия:** `GET /api/nearby?lat&lon&radius_km` и `GET /api/stations?lat1&lon1&lat2&lon2`.
  Станции с ключом **`osm_id`** (совпадает с OSM `amenity=fuel` → легко кросс-мэтчить с Overpass),
  поля: `brand`, `status` ∈ **`yes|queue|low|no`**, `confirmations`, `confirmed`, `last_at`,
  `confidence_base`.
- **CORS:** заголовка `Access-Control-Allow-Origin` **нет** → из браузерного PWA напрямую нельзя;
  в нативном Capacitor зовём через **`CapacitorHttp`** (обходит CORS). В вебе/эмуляторе — символы АЗС
  нейтрального цвета (наличие неизвестно).
- **Заимствуем цветовую семантику статусов:** есть `#39d353` · очередь `#ffd400` · мало `#ff9500` ·
  нет `#ff3b30` · неизвестно `#66ccff` (`FUEL_COLORS` в `js/state.js`).
- **Модель приватности (ориентир для наших треков):** ГдеБЕНЗ не хранит сырые GPS — для отметки «на
  месте» сохраняет только **расстояние** до АЗС. Взять на вооружение в Фазе 5 (запись треков) как
  паттерн «храним производные, а не сырые координаты», где это допустимо.
- **Этика потребления API:** сервис живёт на донаты — обращаться **бережно** (кэш, ограниченный радиус,
  редкие запросы), показывать атрибуцию, не хаммерить. Ленивый вызов только по нажатию ⛽.

## 12. Бэклог / идеи «на потом»

- **ИИ-диалог (LLM)** для планирования и изменения маршрута свободным текстом/голосом — **только онлайн**.
  В движении любое взаимодействие сложнее «да/нет» **запрещено** (принцип №4); полноценный диалог —
  только на остановке. Кандидат в **premium** (Фаза 8).
- Импорт маршрутов (GPX/ссылки), «поделиться треком».
- Групповые поездки (позиции участников).
- Собственный краудсорсинг камер.
- Интеграция с БК мотоцикла (OBD/CAN, если применимо).
- Тёмная/светлая тема автоматически по датчику освещённости.
- Виджеты/быстрый старт «домой»/«на работу» из избранного.
- **Топливный ассистент, развитие:** предупреждение «пора заправиться» по запасу хода (ручной ввод
  расхода/бака) и по расстоянию до следующей АЗС на трассе; фильтр по марке топлива (92/95/98/ДТ);
  отдельный вызов ГдеБЕНЗ `/api/stations` по bbox для плотных городов; кэш наличия на устройстве.
- Калибровка личного `a_lat` по GPX-трекам (curve speed warning, ROADPATH бэклог).
